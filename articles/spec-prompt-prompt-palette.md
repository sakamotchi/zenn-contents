# はじめに

仕様駆動開発のための軽量デスクトップアプリ「**SpecPrompt**」に、新機能として **プロンプトパレット** を追加しました。

https://github.com/sakamotchi/spec-prompt

SpecPrompt自体の概要は以前書いた記事を参照してください。

https://zenn.dev/sakamotchi/articles/spec-prompt-introduction

この記事では、なぜプロンプトパレットを作ったのか、どんな問題を解決するのかを紹介します。

# 解きたかった課題: AI CLIへの「誤送信」

Claude Codeのような対話型AI CLIにプロンプトを入力していると、こんな経験はないでしょうか。

- **複数行のプロンプトを書いている途中で誤って `Enter` を押してしまい、未完成の指示が送信される**
- **ファイルパスを挿入したつもりがターミナルに直接書き込まれ、コマンドの一部として実行されかける**
- **長い指示を推敲したいが、ターミナル上では編集体験が貧弱**

ターミナルは「打ったキーが即座にプロセスに渡る」性質を持つため、推敲しながらプロンプトを組み立てる用途には向いていません。一方で、別のエディタで書いてからコピペする運用は手数が多く、ファイルパスの参照も二度手間になります。

この「**推敲してから明示的に送る**」というワンクッションをアプリ内で完結させるのがプロンプトパレットの役割です。

# プロンプトパレットの基本動作

`Cmd+Shift+P`（macOS）/ `Ctrl+Shift+P`（Win/Linux）でアクティブなターミナルに対してパレットが開きます。

- **Enterは改行**: ターミナルと逆で、Enterは送信ではなく改行として扱う
- **Cmd+Enter / Ctrl+Enter で送信**: 明示的なキー操作で初めてPTYに書き込まれる
- **Esc で閉じる**: 下書きはメモリに保持され、再度開くと復元される

これだけです。「明示的な送信操作を要求する」という一点だけで、誤送信の不安はかなり軽減されます。

ターミナルタブの右クリックメニュー、ターミナル本体の右クリックメニューからも起動できます。

# 起動時のターミナルに送信先を固定する

地味ですが重要な仕様として、**パレットを開いた時点のアクティブターミナルに送信先を固定** します。

パレットを開いたまま別のターミナルタブに切り替えても、送信先は変わりません。ヘッダには「プロンプトを編集 → {タブ名} に送信」と表示されるため、どのターミナルに送るかが一目で分かります。

これは「Aタブで作業しながらBタブ向けの指示を組み立てる」という使い方を想定した挙動です。

# パス挿入との統合

SpecPromptには元々、ファイルパスをターミナルに挿入する3つの経路があります。

1. ファイルツリーの `Cmd+Click`
2. ファイルツリーの右クリックメニュー
3. パスパレット（`Ctrl+P`）でファイルをファジー検索

プロンプトパレットを追加するにあたって、これらすべてを **「パレットが開いていればtextareaのキャレット位置に挿入」** という挙動に切り替えました。

```
パレット閉時: パス → ターミナルPTYに直書き（従来）
パレット開時: パス → パレットtextareaのキャレット位置に挿入
```

つまり、プロンプトを書いている最中に `Ctrl+P` でファイルを検索し、Enterを押すとそのパスが今書いている文章に差し込まれます。選択範囲があれば置換、なければキャレット位置に挿入されます。

仕様書を見ながら「**この実装方針で `src/foo/bar.ts` と `src/foo/baz.ts` を修正してください**」のような指示を、パスを手打ちすることなく組み立てられます。

# 非モーダルで作業を止めない

プロンプトパレットは **非モーダル** です。表示中もファイルツリーやMarkdownプレビューをクリックでき、フォーカス操作も奪いません。

これはVS Codeのコマンドパレットのような「画面全体を覆う」UIではなく、「**書きながら他のペインを参照する**」体験を優先したかったためです。

実装的にはRadix Dialogを `modal={false}` で使い、オーバーレイを `pointer-events: none` にしています。さらに、ツリーから `Cmd+Click` した際にDialogが閉じてしまうのを避けるため、ツリー系セレクタからのフォーカスアウトでは `preventDefault` する処理を入れています。

# タブごとの下書き保持

プロンプトの下書きは **ターミナルタブごとにメモリ上で保持** します。

- パレットを閉じても、同じタブで再度開けば直前の本文が復元される
- 送信に成功すると下書きはクリアされる
- タブを閉じると下書きは破棄される

「途中まで書いたが、ファイルツリーで仕様を確認したくなって閉じた」というケースで、書いた内容が消えないのは想像以上に効きます。

なお、永続化（アプリ再起動後の復元）はv1スコープ外としました。揮発前提のメモリ保持に留めています。

# 細かい配慮

地味ですが、AI CLIとの対話を快適にする上で外せなかった配慮をいくつか挙げます。

## IME二重ガード

日本語入力中の `Enter` や `Cmd+Enter` で送信されないよう、Reactの `compositionstart/end` イベントと `e.nativeEvent.isComposing` の両方で判定しています。ブラウザ実装によってイベントの発火順序が異なるため、片方だけでは取りこぼしが発生します。

## グローバルショートカットの抑止

パレット表示中は、SpecPromptのグローバルショートカット（`Ctrl+Tab` でのペイン切り替え、`Cmd+T` での新規タブ、`F2` でのリネーム等）を抑止します。

allow listには `Ctrl+P`（パスパレット）と `Cmd+Shift+P`（プロンプトパレット自身）だけを残しています。プロンプトを書いている最中にレイアウトが変わると気が散るためです。

## 送信失敗時はトーストで通知

`writePty` が失敗した場合、トーストでエラーを表示しつつ **パレットと本文は維持** します。書いた内容を失わずに再送信できる設計です。

## 挿入時のフラッシュ

パスが挿入された瞬間にtextareaの枠が300msフラッシュします。「いま挿入された」というフィードバックを視覚的に与えるためです。`prefers-reduced-motion: reduce` ではスキップします。

# 典型的な使い方

実際の使い方は次のような流れになります。

1. 仕様書（Markdown）をプレビューで開く
2. `Cmd+Shift+P` でプロンプトパレットを開く
3. 指示を複数行で書く
4. 必要なファイルを `Ctrl+P` で検索してEnter（パスがtextareaに挿入される）
5. ツリーから他のファイルも `Cmd+Click` で追加
6. 最終確認して `Cmd+Enter` で送信

ターミナルの上で恐る恐る指示を打ち、Enterを押す瞬間に「あ、まだ書きかけだった」と慌てる、という体験から解放されます。

# おわりに

プロンプトパレットは、機能としてはとても小さなものです。やっていることは「textareaを開いて、明示的な送信操作で初めてターミナルに書き込む」だけ。

ただ、AI CLIとの対話を毎日続けていると、この **「送信を能動的に決める」** というワンクッションの有無は、体験として大きく変わってきます。誤送信のたびに `Ctrl+C` で打ち切ってやり直す、というストレスがなくなります。

SpecPromptの軽量性（メモリ200MB未満）はそのままに、対話の質を一段上げる機能として追加しました。
