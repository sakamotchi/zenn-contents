# はじめに

運用・保守の現場では、問い合わせチケットが届くたびに「内容を読む → フォルダを作る → 依頼内容を整理する → 調査クエリを書く」という定型作業が発生します。

この記事では、Claude Codeのスキル機能とNotion MCPを組み合わせて、この初動調査を自動化する仕組みを紹介します。

Notion上のチケットURLを渡すだけで、調査に必要なフォルダ・ファイル一式が自動生成されます。

# 背景：運用・保守の初動調査で起きていたこと

運用・保守チームでは、Notionで問い合わせチケットを管理しています。チケットが届くと、担当者は以下の作業を行います：

1. Notionのチケット内容を確認
2. 案件用のフォルダを作成
3. 依頼内容をMarkdownに整理
4. 調査方針を立てる
5. 必要に応じてSQLクエリを準備
6. 必要に応じて監査ログの調査クエリを準備
7. 必要に応じてログ検索クエリを準備

1件あたり15〜30分かかるこの作業が、`/inquiry` スキルで数分に短縮されました。

# 全体像

## 仕組み

```
Notion（チケット管理）
    ↓ Notion MCP（notion-fetch）
Claude Code（/inquiry スキル）
    ↓ 内容分析 → ファイル生成
案件フォルダ/
├── 01_依頼内容.md          ← チケット内容を構造化
├── 02_調査メモ.md          ← 調査テンプレート
├── 03_調査クエリ.sql       ← DB調査SQL（必要な場合）
├── 03_監査ログ調査クエリ.sql ← 監査ログSQL（必要な場合）
└── 04_ログ調査.md          ← ログ検索クエリ（必要な場合）
```

## 使い方

Claude Codeのプロンプトで以下を入力するだけです：

```
/inquiry https://www.notion.so/xxxxxxxx
```

これだけで、Notionのチケット内容を取得・分析し、調査に必要なファイル一式が自動生成されます。

# スキルの設計

## ディレクトリ構成

```
.claude/skills/
├── inquiry/SKILL.md           # 初動対応スキル
├── db-schema/SKILL.md         # DBスキーマ検索スキル（メインDB）
├── kouji-db-schema/SKILL.md   # 別システムのスキーマ検索
└── seikyu-db-schema/SKILL.md  # 別システムのスキーマ検索
```

`/inquiry` スキルは、SQL作成時に `/db-schema` スキルを内部的に呼び出す連携構造になっています。

## SKILL.mdの構造

スキル定義は以下のようなフロントマターとMarkdownで記述します：

```markdown
---
name: inquiry
description: お問い合わせの初動対応。NotionのURLから内容を取得し、
  整理されたフォルダとファイルを作成する。
---

# お問い合わせ初動対応スキル

## 引数
$ARGUMENTS にはNotionのページURLを指定してください。

## 手順
以下の手順を順番に実行してください。

### 1. Notion内容の取得
...
### 2. 内容の分析
...
### 3. フォルダの作成
...
```

以降、各ステップの設計を解説します。

# ステップ1：Notionからのチケット取得

## Notion MCPの活用

Claude CodeにはMCP（Model Context Protocol）サーバーを接続する機能があります。Claudeチャットアプリのコネクタ機能でNotionを有効化すると、Claude Codeから直接Notionの内容を取得できます。

```markdown
### 1. Notion内容の取得

- Notion MCP (`notion-fetch`) を使って、
  指定されたNotionページの内容を取得してください。
- 取得できない場合はユーザーにエラーを報告してください。
```

Notion MCPのセットアップは以下の手順です：

1. Claudeチャットアプリを開く
2. 設定 → 「コネクタ」セクション → Notionを有効化
3. OAuthで認証を完了
4. Claude Codeを再起動

これでスキル内から`notion-fetch`が利用可能になります。

# ステップ2：内容の分析と添付ファイル対応

## チケット内容の構造化

取得したNotionページから以下の情報を抽出するよう指示しています：

```markdown
### 2. 内容の分析

取得したNotionページの内容を分析し、以下の情報を抽出してください:

- **タイトル**: お問い合わせの簡潔なタイトル（フォルダ名に使用）
- **依頼内容**: 何を依頼されているか
- **質問内容**: 具体的な質問事項
- **関連するシステム/機能**: どの部分に関係するか
- **緊急度・優先度**: 記載があれば抽出
- **添付ファイル**: ファイル添付（Excel、PDF、画像、CSV等）があるか確認
```

## 添付ファイルの対応

Notion MCPはファイルのダウンロードに対応していないため、添付ファイルがある場合はユーザーに手動ダウンロードを依頼する設計にしています：

```markdown
### 2-1. 添付ファイルの確認とダウンロード依頼

**添付ファイルがある場合:**

1. ユーザーに以下を伝えて手動ダウンロードを依頼する:
   - 検出された添付ファイルの一覧
   - 「Notion MCPではファイルのダウンロードができないため、
     手動でダウンロードをお願いします」
   - ダウンロード先フォルダのパスを案内

2. ユーザーがファイルを配置したら、以下のツールで解析する:
   - **Excel (.xlsx)**: Python（pandas / openpyxl）で解析
   - **PDF**: Readツールで内容を読み取り
   - **画像**: Readツールで視覚的に確認
   - **CSV**: pandas またはReadツールで読み取り
```

この設計のポイントは、MCPの制約（ファイルダウンロード不可）を正直にユーザーに伝え、手動ダウンロード後の解析は自動化する点です。Python仮想環境にpandasとopenpyxlを用意しておくことで、Excelファイルの解析もClaude Codeに任せられます。

# ステップ3：フォルダとファイルの自動生成

## フォルダの命名規則

```markdown
### 3. フォルダの作成

- `yyyymmdd-タイトル` の形式でフォルダを作成
- 日付は本日の日付を使用
- タイトルはNotionの内容から適切に要約した短い日本語

例: 20260219-メーター表示不具合/
```

`.gitignore`で`20[0-9][0-9][0-9][0-9][0-9][0-9]-*/`パターンを除外することで、案件フォルダはリポジトリに含めない設計にしています。

## 生成されるファイル

### `01_依頼内容.md` - チケット内容の構造化

```markdown
# 依頼内容

## 概要
（お問い合わせの概要を1-2文で）

## 依頼内容の詳細
（依頼の詳細を箇条書きで整理）

## 質問事項
（質問内容を箇条書きで整理）

## 添付ファイル
（添付ファイルがある場合に記載）

## 関連情報
- Notion URL: （元のURL）
- 関連するシステム/機能: （特定できた場合）
- 受付日: yyyy/mm/dd
```

Notionのフリーフォーマットな記述が、調査しやすい構造に自動変換されます。

### `02_調査メモ.md` - 調査テンプレート

```markdown
# 調査メモ

## 調査方針
（依頼内容をもとに、どのような調査が必要かを記載）

## 調査結果
（調査を進めながら記載）

## 原因
（判明した場合に記載）

## 対応方針
（対応方針が決まったら記載）
```

スキルの実行時点で、依頼内容に基づいた調査方針の初期案が自動記載されます。

# ステップ4：SQLクエリの自動生成

## 要否判断の基準

スキル定義内でSQLクエリが必要かどうかの判断基準を明示しています：

```markdown
### 5. SQLクエリの要否判断と作成

**SQLクエリが必要と判断する基準:**
- 特定のユーザー、施設、設備などのデータ確認が必要な場合
- エラーの原因調査でDBの状態確認が必要な場合
- データの整合性チェックが必要な場合
- 特定のレコードの存在確認や件数確認が必要な場合
```

## DBスキーマスキルとの連携

SQLクエリ作成時の重要なルールとして、**カラム名を推測してSQLを書いてはいけない**というルールを設けています。

```markdown
- **SQLで使用する全テーブルについて、必ず `/db-schema テーブル名`
  スキルでカラム情報を確認してからクエリを作成すること。**
  カラム名を推測してSQLを書いてはいけない。
```

`/db-schema` スキルは、あらかじめDockerコンテナからエクスポートしたスキーマリファレンス（Markdown形式）を検索するスキルです。

```
/db-schema inspections          # テーブル名で完全一致検索
/db-schema meter                # 部分一致検索
/db-schema column:building_id   # カラム名検索
/db-schema refresh              # 最新スキーマの再エクスポート
```

このスキル連携により、AIがカラム名を間違えて無効なSQLを生成する問題を防いでいます。

# ステップ5：監査ログ調査の自動生成

運用・保守では「いつ・誰がデータを変更したか」の追跡が頻繁に求められます。スキルでは監査ログ調査の要否判断と、調査クエリの自動生成を行います。

## 要否判断の基準

```markdown
**監査ログ調査が必要と判断する基準:**
- データが「いつ」「誰によって」変更・削除されたかの特定が必要な場合
- 「紐づけが外れた」「データが消えた」「設定が変わった」等、
  意図しないデータ変更の調査
- 物理削除が行われた可能性がある場合
- 特定期間内の操作履歴を時系列で確認する必要がある場合
```

## 監査ログテーブルの構造をスキルに埋め込む

ここがこのスキルの重要なポイントです。監査ログのテーブル構造をスキル定義内に直接埋め込んでいます。

```markdown
**監査ログは2つのテーブルに分かれている:**

| テーブル | 記録対象 | 用途 |
|----------|---------|------|
| ledgers | Eloquent Model 経由の操作 | Modelのcreated/updated/deletedを記録 |
| ledger_renewals | 全SQL操作 | バルクINSERT/DELETE、生SQL等も記録 |
```

なぜ監査ログのスキーマを `/db-schema` スキルに任せず、直接埋め込んでいるかというと、監査ログのクエリは定型的なパターンが決まっているためです。テーブル構造とクエリテンプレートをセットでスキルに埋め込むことで、より正確なクエリが生成されます。

## パーティションキーの制約

監査ログテーブルの1つは年月（`ym`）をパーティションキーとしています。パーティションキーの条件を指定し忘れると、クエリのパフォーマンスが極端に悪化します。

```markdown
| ym | int unsigned | **パーティションキー**。
  必ず `=` か `IN` で検索条件に含めること |
```

このような「必ず守るべき制約」をスキル定義に明記することで、AIが制約を無視したクエリを生成するリスクを減らしています。

## 両テーブルの検索を強制

```markdown
**両方のテーブルに対してクエリを作成すること。**
特にCSV一括アップロードやバッチ処理など、
Modelを経由しない操作はledgersに記録されず
ledger_renewalsにのみ記録される。
```

片方のテーブルだけ検索して「該当なし」と結論づけてしまうミスを防ぐため、両テーブルの検索を明示的に強制しています。

# ステップ6：ログ調査の自動生成

エラー調査やパフォーマンス問題の場合、CloudWatchやDatadogでのログ調査が必要になります。

## APIエンドポイントの特定

ログ検索の精度を上げるため、まずフロントエンドのソースコードやAPIルート定義から関連するエンドポイントを特定します。

```markdown
#### 7-1. APIエンドポイントの特定

1. 依頼内容から関連する機能・画面・操作を抽出
2. フロントエンドのソースから該当画面のAPI呼び出しを検索
3. APIルート定義から対応するエンドポイントを確認
```

## ログ検索クエリのテンプレート

特定したエンドポイントを使って、CloudWatchやDatadogの検索クエリを生成します。

```markdown
| 情報 | CloudWatch クエリ例 | Datadog クエリ例 |
|------|---------------------|------------------|
| ユーザー特定 | `{ $.context.user_id = "12345" }` | `@user_id:12345` |
| エラー検索 | `{ $.level = "ERROR" }` | `status:error` |
| エンドポイント特定 | `{ $.url = "*endpoint*" }` | `@http.url:*endpoint*` |
```

検索クエリの構文はサービスごとに異なりますが、スキルにテンプレートを用意しておくことで、すぐに使えるクエリが生成されます。

# スキル設計のポイント

このスキルの設計で工夫した点をまとめます。

## 1. 要否判断の基準を明示する

各ステップで「必要と判断する基準」を明確に記載しています。これにより、AIが不要なファイルを生成したり、必要なファイルを生成し忘れたりすることを防ぎます。

```markdown
**SQLクエリが必要と判断する基準:**
- 特定のユーザーや設備のデータ確認が必要な場合
- エラーの原因調査でDBの状態確認が必要な場合
...

**SQLクエリが不要と判断した場合:**
`02_調査メモ.md` の調査方針にその旨を記載してください。
```

「不要な場合」の対応も明記することで、判断の抜け漏れを防いでいます。

## 2. スキル間の連携を活用する

`/inquiry` スキルの中で `/db-schema` スキルを呼び出す設計にしています。スキルは単体で完結させるだけでなく、他のスキルと組み合わせることで、より正確な出力を得られます。

## 3. MCPの制約に対応する

Notion MCPにはファイルダウンロードの制約があります。制約を隠すのではなく、「ユーザーに手動ダウンロードを依頼する」というステップを明示的にスキルに組み込んでいます。

## 4. ドメイン知識をスキルに埋め込む

監査ログのテーブル構造やパーティションキーの制約など、ドメイン固有の知識をスキル定義に埋め込んでいます。これにより、AIが毎回スキーマを調べる必要がなく、正確なクエリを素早く生成できます。

## 5. .gitignoreで案件フォルダを除外する

案件フォルダには個別の調査内容が含まれるため、`.gitignore`で除外しています。一方、スキル定義やステアリングドキュメントはGit管理してチームで共有しています。

```gitignore
20[0-9][0-9][0-9][0-9][0-9][0-9]-*/   # 個別案件フォルダ
venv/                                    # Python仮想環境
.claude/settings.local.json              # 個人の権限設定
```

# チームでの共有

このワークスペースはチームで共有する設計にしています。

## 共有するもの

| 対象 | 内容 |
|------|------|
| `CLAUDE.md` | プロジェクト指示（環境URL、リポジトリ構成、共通ルール） |
| `.claude/skills/` | カスタムスキル定義 |
| `docs/steering/` | アーキテクチャ、トラブルシューティング等のドキュメント |
| `docs/*_schema_reference.md` | DBスキーマリファレンス |

## 共有しないもの（.gitignore）

| 対象 | 理由 |
|------|------|
| `20XXXXXX-*/` | 個別案件の作業フォルダ |
| `.claude/settings.local.json` | 個人のMCP権限設定 |
| `venv/` | Python仮想環境（requirements.txtから再構築可能） |

スキル定義をチーム共有することで、メンバー全員が同じ品質の初動調査を行えます。

# 導入のステップ

自分のプロジェクトに導入する場合の手順です。

## 1. 保守ワークスペースの作成

```bash
mkdir maintenance && cd maintenance
git init
```

## 2. CLAUDE.mdの作成

プロジェクトの概要、関連リポジトリのパス、共通ルールを記載します。

## 3. Notion MCPの設定

Claudeチャットアプリのコネクタ機能でNotionを有効化します。

## 4. スキルの作成

`.claude/skills/inquiry/SKILL.md` を作成し、自分のプロジェクトに合わせた手順を定義します。

ポイントは：
- **手順を具体的に書く**: 曖昧な指示ではなく、生成するファイルのフォーマットまで明記する
- **判断基準を明示する**: 「SQLが必要な場合」「不要な場合」の両方を定義する
- **ドメイン知識を埋め込む**: 監査ログのスキーマなど、プロジェクト固有の情報を含める
- **制約を明記する**: パーティションキーの制約など、守るべきルールを書く

## 5. DBスキーマスキルの作成（任意）

SQLクエリの自動生成を行う場合、DBスキーマを検索するスキルがあると精度が大幅に向上します。スキーマリファレンスをMarkdown形式でエクスポートし、検索スキルを定義します。

# まとめ

Claude Codeのスキル機能とNotion MCPを組み合わせることで、運用・保守の初動調査を自動化できました。

1. **Notion MCP**でチケット内容を自動取得
2. **スキルの手順定義**で構造化されたファイルを自動生成
3. **スキル間連携**（inquiry → db-schema）でSQLの正確性を担保
4. **ドメイン知識の埋め込み**で監査ログクエリの品質を向上
5. **チーム共有**でメンバー全員が同じ品質の初動調査を実現

定型的な作業をスキルに任せることで、エンジニアは「調査結果の分析」「原因の特定」「対応方針の決定」という、より本質的な作業に集中できるようになります。
